## Purpose

The objective of this microservice is to provide **Identity & Access Management** capabilities for the SD-WAN platform, acting as the backend adapter/orchestrator around **Keycloak** for:

- user group (realm/tenant) lifecycle management
- user lifecycle management inside a user group
- authentication flows, including **optional MFA** (Email OTP / SMS OTP / TOTP)

In that sense, this microservice exposes a set of operations that can be used by upstream consumers (e.g. sdwan-portal-api) to manage tenants/users and authenticate users in a consistent way.

### Auth

This feature exists to give support for upstream API consumers whenever they want to perform authentication operations against the identity provider (Keycloak), including MFA orchestration.

In the next subsections we will be describing this module use cases.

* **Login (username/password with optional MFA)**

Through this use case, the API consumer is able to perform an initial login attempt using username and password.  
Depending on configuration and user state, the response indicates whether MFA is required, what methods are available, and what the user still needs to configure.

**[POST]** _/v1/auth/login_

```plantuml
!theme bluegray

actor Consumer as actor
participant "sdwan-portal-api" as controller
participant "sdwan-identity-access-api" as service
participant "Keycloak Admin API" as kcAdmin
participant "Keycloak Token (Direct Grant)" as kcToken
participant "PendingMfaStore" as store
participant "JwtService" as jwt

activate actor
actor -> controller : POST /v1/auth/login (username, password, user_group_id, mfaEnabled?)
activate controller
controller -> service : authLogin(LoginRequestApiV1, mfaEnabled)
activate service

service -> kcAdmin : resolve realm by user_group_id\n(search realms, pick realm name)
kcAdmin --> service : realmName / RealmResource
service -> kcAdmin : search user by username
kcAdmin --> service : userId (or not found)

alt user not found
  service --> controller : HTTP 404 USER_NOT_FOUND
else user found
  service -> kcToken : validateCredentialsDetailed(realm, username, password)\n(using direct grant)
  kcToken --> service : ok / error details

  alt invalid password / login failed
    service --> controller : HTTP 401 INVALID_PASSWORD or LOGIN_FAILED
  else credentials ok
    alt MFA disabled or not required
      service -> jwt : issue(realm, userId, username)
      jwt --> service : token
      service --> controller : HTTP 200 (mfaRequired=false, token)
    else MFA required
      service -> store : create loginAttemptId\n(store realm/userId/username/(password if needed))
      store --> service : loginAttemptId
      service --> controller : HTTP 200 (mfaRequired=true,\nloginAttemptId, methods, needs)
    end
  end
end

deactivate service
controller -> actor : HTTP response
deactivate controller
deactivate actor
```

<br>

The request consists of:

* userGroupId (tenant/realm identifier)
* username
* password
* optional query parameter: mfaEnabled (defaults to true)

In order to get to this result the following steps are performed:

1. We resolve the Keycloak realm based on the provided user group id
2. We locate the user inside that realm by username
3. We validate credentials using Keycloak direct grant
4. If MFA is disabled (or not required), we issue a final JWT and return it
5. Otherwise, we create and return a loginAttemptId and indicate which MFA methods can be used and which setup steps are still required

* **MFA Email OTP - Send**

Through this use case, the API consumer requests the system to send a one-time password to the user email address (stored in Keycloak).

**[POST]** _/v1/auth/mfa/email/send_

```plantuml
!theme bluegray

actor Consumer as actor
participant "sdwan-portal-api" as controller
participant "sdwan-identity-access-api" as service
participant "PendingMfaStore" as store
participant "Keycloak Admin API" as kcAdmin
participant "EmailSender" as email

activate actor
actor -> controller : POST /v1/auth/mfa/email/send (loginAttemptId)
activate controller
controller -> service : authMfaEmailSend(loginAttemptId)
activate service

service -> store : get(loginAttemptId)
alt invalid attempt
  service --> controller : HTTP 400 INVALID_ATTEMPT
else valid attempt
  service -> kcAdmin : read user (realm, userId)\nget email
  kcAdmin --> service : email (or missing)
  alt no email
    service --> controller : HTTP 400 NO_EMAIL_ON_ACCOUNT
  else email ok
    service -> store : setEmailOtp(loginAttemptId, code, exp)
    service -> email : sendOtp(email, code)
    email --> service : ok
    service --> controller : HTTP 200 {sent:true}
  end
end

deactivate service
controller -> actor : HTTP response
deactivate controller
deactivate actor
```

* **MFA Email OTP - Verify**

Through this use case, the API consumer verifies the OTP code sent by email.  
On success, a final JWT token is issued.

**[POST]** _/v1/auth/mfa/email/verify_

```plantuml
!theme bluegray

actor Consumer as actor
participant "sdwan-portal-api" as controller
participant "sdwan-identity-access-api" as service
participant "PendingMfaStore" as store
participant "JwtService" as jwt

activate actor
actor -> controller : POST /v1/auth/mfa/email/verify (loginAttemptId, code)
activate controller
controller -> service : authMfaEmailVerify(loginAttemptId, code)
activate service

service -> store : get(loginAttemptId)
alt invalid attempt
  service --> controller : HTTP 400 INVALID_ATTEMPT
else valid attempt
  service -> store : consumeValidEmailOtp(loginAttemptId, code)
  alt invalid/expired otp
    service --> controller : HTTP 400 INVALID_OR_EXPIRED_OTP
  else otp ok
    service -> jwt : issue(realm, userId, username)
    jwt --> service : token
    service -> store : remove(loginAttemptId)
    service --> controller : HTTP 200 FinalTokenResponseApiV1(token)
  end
end

deactivate service
controller -> actor : HTTP response
deactivate controller
deactivate actor
```

* **MFA TOTP - Callback**

Through this use case, Keycloak redirects back to the backend callback endpoint after the browser flow (authorization code flow).  
The backend optionally exchanges the authorization code for a token (PoC), and then redirects the browser onward.

**[GET]** _/v1/auth/mfa/totp/callback_

```plantuml
!theme bluegray

actor Browser as actor
participant "Keycloak" as kc
participant "sdwan-identity-access-api" as service
participant "PendingMfaStore" as store
participant "Keycloak Token Endpoint" as kcToken

activate Browser
Browser -> kc : User completes CONFIGURE_TOTP
kc -> Browser : redirect to /v1/auth/mfa/totp/callback?code=...&state=...
Browser -> service : GET /v1/auth/mfa/totp/callback (code, state)
activate service

service -> store : lookup/bind state (optional)
store --> service : loginAttemptId (optional)

opt exchange code for token (PoC)
  service -> kcToken : POST /token (authorization_code)
  kcToken --> service : HTTP 200/4xx (ignored in PoC)
end

service -> store : cleanup state
service --> Browser : HTTP 302 FOUND (redirect)
deactivate service
deactivate Browser
```

* **MFA TOTP - Verify**

Through this use case, the API consumer verifies a TOTP code by performing a Keycloak direct grant request that includes the TOTP code.  
On success, a final JWT token is issued.

**[POST]** _/v1/auth/mfa/totp/verify_

```plantuml
!theme bluegray

actor Consumer as actor
participant "sdwan-portal-api" as controller
participant "sdwan-identity-access-api" as service
participant "PendingMfaStore" as store
participant "Keycloak Token (Direct Grant + TOTP)" as kcToken
participant "JwtService" as jwt

activate actor
actor -> controller : POST /v1/auth/mfa/totp/verify (loginAttemptId, code)
activate controller
controller -> service : authMfaTotpVerify(loginAttemptId, code)
activate service

service -> store : get(loginAttemptId)
alt invalid attempt
  service --> controller : HTTP 400 INVALID_ATTEMPT
else valid attempt
  alt password missing in attempt
    service --> controller : HTTP 400 PASSWORD_MISSING
  else password present
    service -> kcToken : validateCredentialsWithTotp(realm, username, password, totpCode)
    kcToken --> service : ok / invalid
    alt invalid totp
      service --> controller : HTTP 400 INVALID_TOTP
    else ok
      service -> jwt : issue(realm, userId, username)
      jwt --> service : token
      service -> store : remove(loginAttemptId)
      service --> controller : HTTP 200 FinalTokenResponseApiV1(token)
    end
  end
end

deactivate service
controller -> actor : HTTP response
deactivate controller
deactivate actor
```

## Dependencies

This microservice runtime dependencies are illustrated in the following diagram:

```plantuml
!theme bluegray

package "OSS LAN" {
  node SDWAN {
    HTTP - [sdwan-identity-access-api]
    [sdwan-portal-api]
  }
  node IDENTITY {
    [Keycloak]
  }
  node NOTIFICATION {
    [SMTP / Email Provider]
    [SMS Provider]
  }
}

[sdwan-portal-api] --> [sdwan-identity-access-api]
[sdwan-identity-access-api] --> [Keycloak]
[sdwan-identity-access-api] --> [SMTP / Email Provider]
[sdwan-identity-access-api] --> [SMS Provider]
```

As can be seen in the image above, the dependencies include:

* Keycloak: identity provider used for realms (user groups), users, required actions and authentication validation (direct grant / browser flows)
* SMTP / Email provider: used to send OTP and/or Keycloak action emails (depending on environment configuration)
* SMS provider: used to send OTP codes by SMS
* Upstream consumer (e.g. sdwan-portal-api): uses this microservice as the single entry point for identity/tenant/user operations and MFA orchestration
